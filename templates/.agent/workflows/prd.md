# /prd - Product Requirements Document Workflow

## Overview

Use this workflow to draft or refine a Product Requirements Document (PRD) before implementation starts.

## When to Use

- Starting a new feature or product initiative
- Clarifying vague requirements into actionable scope
- Aligning product, engineering, and design on outcomes
- Defining acceptance criteria and delivery boundaries

## Workflow

### Step 1: Define Problem and Outcome

- Problem statement
- Target users and jobs-to-be-done
- Business outcome and success metrics

### Step 2: Define Scope

- In-scope capabilities
- Out-of-scope boundaries
- Constraints and assumptions

### Step 3: Capture Functional Requirements

- User stories or requirement statements
- Priority (must, should, could)
- Dependencies and risks

### Step 4: Define Non-Functional Requirements

- Performance expectations
- Security and compliance requirements
- Accessibility and reliability targets

### Step 5: Acceptance Criteria and Validation

- Testable acceptance criteria
- Definition of done
- Rollout and measurement plan

## Output

- [ ] Problem and goals defined
- [ ] Scope and constraints documented
- [ ] Functional and non-functional requirements captured
- [ ] Acceptance criteria testable and complete
